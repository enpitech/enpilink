enpilink can authenticate **end users** so your tools know who is calling and
you can gate sensitive tools behind sign-in. It is **opt-in and off by
default** — until you enable it, `/mcp` stays open exactly as before and
existing apps are unaffected.

<Note>
enpilink works in **two modes**, both opt-in:

1. **Resource-server only** — point enpilink at an existing OAuth authorization
   server via its issuer + JWKS URL. enpilink validates tokens and enforces
   per-tool scopes (the foundation; see [The model](#the-model)).
2. **Co-hosted Authorization Server** (recommended for real hosts) — enpilink
   *also* hosts the OAuth endpoints + a **branded login page** and proxies the
   actual credential check to a configurable **upstream identity provider**
   (Google, GitHub, Auth0, your own). The host discovers and completes OAuth
   against enpilink; every successful sign-in is **tracked** (see
   [Co-hosting the Authorization Server](#co-hosting-the-authorization-server)).
</Note>

## The model

Following the [MCP Authorization spec](https://modelcontextprotocol.io/) and
OAuth 2.1, your enpilink server is an **OAuth Resource Server**:

- The chat host (ChatGPT, Claude, …) is the OAuth **client**. It obtains an
  access token from your **authorization server** and attaches it as a
  `Bearer` token on each `/mcp` request.
- enpilink **validates** that token (signature via JWKS, plus `iss` / `aud` /
  `exp`), then enforces each tool's [`securitySchemes`](/api-reference/register-tool#securityschemes).
- A tokenless request to a protected tool gets **401** with a
  `WWW-Authenticate` challenge carrying `resource_metadata=` — pointing the
  host at your [RFC 9728 Protected Resource Metadata](https://datatracker.ietf.org/doc/html/rfc9728)
  document (served at `/.well-known/oauth-protected-resource`), which lists the
  authorization server to sign in at.

`/mcp` is **not** route-locked. A token is validated *when present*; gating
happens **per tool**, so public (`noauth`) and protected tools coexist on the
same endpoint. This is the seam guest / lazy auth builds on.

## Enabling it

The minimal config is the authorization server's `issuer`, the `audience` your
tokens are bound to (usually your public `/mcp` URL — RFC 8707), and the
`jwksUrl` to verify signatures.

```typescript
const server = new McpServer(
  { name: "my-app", version: "1.0.0" },
  undefined,
  {
    auth: {
      enabled: true,
      issuer: "https://auth.example.com",
      audience: "https://my-app.example.com/mcp",
      jwksUrl: "https://auth.example.com/.well-known/jwks.json",
    },
  },
);
```

Or entirely via environment (no code change):

```bash
ENPILINK_AUTH=1
ENPILINK_AUTH_ISSUER=https://auth.example.com
ENPILINK_AUTH_AUDIENCE=https://my-app.example.com/mcp
ENPILINK_AUTH_JWKS_URL=https://auth.example.com/.well-known/jwks.json
```

See the full key/env table in the [`McpServer` reference](/api-reference/mcp-server#built-in-end-user-auth-opt-in).

## Gating tools

Declare each tool's auth requirement with
[`securitySchemes`](/api-reference/register-tool#securityschemes). When auth is
enabled these are **enforced automatically** — no handler boilerplate:

```typescript
server
  // Public — runs without a token, even when auth is on.
  .registerTool(
    { name: "search-public", securitySchemes: [{ type: "noauth" }] },
    async () => ({ content: "anyone can call me" }),
  )
  // Protected — requires a valid token carrying the `orders.read` scope,
  // else 401 (no token) / 403 insufficient_scope (missing scope).
  .registerTool(
    {
      name: "my-orders",
      securitySchemes: [{ type: "oauth2", scopes: ["orders.read"] }],
    },
    async (_args, extra) => {
      const id = getAuthInfo(extra);
      return { content: `Orders for ${id?.sub}` };
    },
  );
```

## Reading the signed-in user

The validated principal reaches handlers via `extra.authInfo`. Use the typed
[`getAuthInfo`](/api-reference/mcp-server#getauthinfo) helper:

```typescript
import { getAuthInfo } from "enpilink/server";

async (_args, extra) => {
  const id = getAuthInfo(extra); // undefined for guests
  // id?.sub     — stable per-user id (the JWT `sub` claim)
  // id?.scopes  — granted scopes
  // id?.claims  — the full token claim set (email, etc.)
};
```

The `sub` claim is your stable per-user tracking key.

## Custom verifiers

The built-in verifier ([`createJwtVerifier`](/api-reference/mcp-server#createjwtverifier))
validates standard JWTs against a JWKS URL. To use token introspection or a
provider SDK instead, pass your own `verifier` (any `OAuthTokenVerifier`) in
the `auth` config — it overrides the built-in one.

## Co-hosting the Authorization Server

In resource-server-only mode the host still has to find an authorization server
to sign in at. With a **co-hosted Authorization Server**, enpilink *is* that
server: it serves the OAuth discovery + endpoints, shows a **branded login
page**, and delegates the real credential check to a configurable **upstream
identity provider**. This is the standards-based (MCP Authorization spec) path
designed for real chat hosts (ChatGPT, Claude, …) and it lets enpilink
**track every sign-in**.

<Note>
The end-to-end OAuth loop with a live ChatGPT/Claude connector has **not yet
been verified in production** — this follows the MCP Authorization spec
(2025-11-25) and is designed to interoperate with spec-compliant hosts. If you
encounter an issue completing the loop from a real host, please open an issue.
Use a JWT-issuing upstream (Auth0, Okta, Entra ID) for the most reliable
token-validation path; Google/GitHub's opaque-token flows require an upstream
that re-issues JWTs (not raw access tokens) for enpilink's JWKS verifier.
</Note>

### The login flow

```
host (ChatGPT/Claude)
  │  1. tokenless call → 401 + WWW-Authenticate (resource_metadata=)
  │  2. discovers our AS via /.well-known/oauth-protected-resource
  │     → /.well-known/oauth-authorization-server
  ▼
enpilink /authorize  ──3. branded page ("Sign in" OR "Continue as guest")──▶ user
  │  4a. "Sign in" → redirect to the UPSTREAM provider's /authorize
  │  4b. "Continue as guest" → enpilink mints a guest code (no upstream) → host callback
  ▼
upstream IdP (Google/GitHub/Auth0/…)  ──5. user authenticates──▶
  │  6. redirect to enpilink's /authorize/callback with ?code=… (federating)
  ▼
enpilink /token  ──7. PROXY mode: proxies the exchange upstream
  │             FEDERATING mode: validates PKCE locally, then MINTS + SIGNS
  │             enpilink's own token (bound to the resolved real/guest `sub`)
  │  8. returns the token to the host
  ▼
host attaches the token to every /mcp request
enpilink validates it (JWKS) and RECORDS a session keyed by `sub`
```

In **federating** mode the upstream returns to enpilink's own
`/authorize/callback` (not the host) so enpilink can resolve the user and mint
its own token; the host only ever sees the enpilink-issued token.

The branded page is **server-rendered and out-of-band** (the browser tab the
host opens), not the chat iframe. It is enpilink-teal and consistent with the
console; deeper customization arrives with the Auth tab.

### Login-flow security model (state binding + cookie + redirect allowlist)

The federation flow is hardened against authorization-code injection, CSRF, and
open-redirect:

- **Server-side pending state + opaque id.** The authorization parameters
  (`redirect_uri`, PKCE `code_challenge`, scopes, host `state`) are stored
  **server-side**, keyed by a random opaque id, with a short TTL (10 min) and
  **single-use** semantics. Only that opaque id is ever placed in the upstream
  OAuth `state` — the parameters are never serialized into a URL the browser
  carries, so they cannot be forged or tampered with.
- **Browser binding via an HttpOnly cookie.** When the flow starts, enpilink
  sets a short-lived `HttpOnly; SameSite=Lax; Secure` cookie holding that id.
  `/authorize/callback` and `/authorize/guest` require **both** the cookie and
  the `state`/`id` query param, and they must be **equal**, before the stored
  pending state is looked up and consumed. A missing/mismatched cookie, or an id
  with no matching (or expired/already-used) server-side entry, is rejected with
  `400`. This blocks cross-session replay and CSRF (an attacker cannot set an
  `HttpOnly` cookie in the victim's browser).
- **`redirect_uri` allowlist.** The effective `redirect_uri` is validated
  against the registered `ENPILINK_AUTH_REDIRECT_URIS` at **every** step before
  any redirect — unregistered URIs are rejected (code-exfiltration / open
  redirect prevention).

<Note>
The `Secure` cookie attribute is set automatically when the public issuer
(`ENPILINK_AUTH_ISSUER`) is `https://…` (it works behind a tunnel and locally
over `http`). The pending-auth store is **in-memory** for the single-instance
model — a multi-instance deployment needs a shared store (or sticky sessions),
the same caveat as the in-memory code/refresh state.
</Note>

### Two provider modes

enpilink picks one of two Authorization-Server provider modes, **based on
whether a token signing key is set**:

| Mode | When | Who issues the token | Enables |
| --- | --- | --- | --- |
| **Transparent proxy** | upstream set, **no** `ENPILINK_AUTH_SIGNING_KEY` | the **upstream** IdP | logged-in flow only |
| **Federating issuer** | upstream set **and** `ENPILINK_AUTH_SIGNING_KEY` set | **enpilink** (signs its own tokens) | logged-in **+ guest + lazy/step-up** |

- **Transparent proxy** (`ProxyOAuthServerProvider`): tokens are issued by the
  upstream provider; enpilink validates them via the upstream JWKS (the same
  [`createJwtVerifier`](#custom-verifiers) as resource-server mode). enpilink is
  **not** a token issuer — the smallest security surface. Point
  `ENPILINK_AUTH_JWKS_URL` at the **upstream** JWKS.
- **Federating issuer** (`FederatingOAuthProvider`): enpilink **mints + signs
  its own tokens** with a deterministic Ed25519 keypair derived from
  `ENPILINK_AUTH_SIGNING_KEY`, advertises them at its own
  `/.well-known/jwks.json`, and validates them locally. It still federates the
  *login* to the upstream IdP, but it can also issue **guest tokens** (no
  upstream round-trip). Point `ENPILINK_AUTH_ISSUER`/JWKS at **enpilink itself**
  (the default — enpilink serves the JWKS).

<Note>
Setting `ENPILINK_AUTH_SIGNING_KEY` is what turns on **guest mode** and the
federating issuer. Without it you get the (still fully supported) proxy mode.
The signing key is **env-only** and must be a strong, stable secret — every
token enpilink issues is signed with it.
</Note>

### Guest mode & lazy / step-up auth

In federating mode the branded entry page offers **"Continue as guest"**
alongside **"Sign in"**:

- **Guest** mints a **limited guest token** with its own short scopes
  (`guest`) and a subject of `guest:<random-id>` — the `guest:` prefix means a
  guest `sub` can **never collide** with a real upstream `sub`. Guests are
  tracked coarsely; durable identity requires a real login. A guest token is
  **never** granted `oauth2`-scoped tools.
- **Lazy / step-up:** `/mcp` stays open on connect (no forced login).
  `{ type: "noauth" }` tools run for **anyone** — true anonymous (no token) and
  guests alike. The first time a caller hits an `{ type: "oauth2" }` tool
  without a sufficient token (tokenless **or** guest), enpilink returns a
  **401/403 + `WWW-Authenticate`** challenge, which triggers the host's OAuth
  out-of-band. After the user signs in, the **same tool call succeeds** and the
  handler sees the real `sub`. Use the `isGuestSub(sub)` helper (or the `guest`
  claim) to branch behavior.

```typescript
import { getAuthInfo, isGuestSub } from "enpilink/server";

server.registerTool(
  { name: "save", inputSchema: {}, securitySchemes: [{ type: "oauth2", scopes: ["write"] }] },
  async (args, extra) => {
    const sub = getAuthInfo(extra)?.sub;        // real `sub` here (guests are challenged first)
    if (isGuestSub(sub)) { /* never reached for an oauth2 tool */ }
    return { content: `saved for ${sub}` };
  },
);
```

## Reading the user in your view

The view runs in an **identity-blind iframe** — the host passes it the locale,
theme, and tool I/O, but **no user identity**. So a view learns "who am I" via a
**tool round-trip**: it calls a small built-in tool, and the server (which has
`extra.authInfo`) answers with the caller's identity.

When auth is enabled, enpilink **auto-registers** a `{ type: "noauth" }` tool
named **`enpilink_whoami`**. It is `noauth` on purpose — anonymous and guest
callers must be able to call it (that is how the view discovers it is *not*
signed in). It returns only identity/claims and **never a token**.

### `useAuth()`

The [`useAuth`](/api-reference/use-auth) hook does the round-trip for you and
models the **three states** explicitly:

```tsx
import { useAuth } from "enpilink/web";

function Account() {
  const { state, name, email, isGuest, isLoading, refresh } = useAuth();
  if (isLoading) return <Spinner />;
  switch (state) {
    case "authed":    return <p>Signed in as {name ?? email}</p>;
    case "guest":     return <p>Browsing as a guest</p>;
    case "anonymous": return <SignInButton onSignedIn={refresh} />;
  }
}
```

| State | Meaning |
| --- | --- |
| `anonymous` | No token at all (or auth is off / the tool isn't present — `useAuth` degrades to this and **never throws**). |
| `guest` | A minted guest token (the `sub` carries the `guest:` prefix). |
| `authed` | A real, upstream-backed signed-in user. |

`useAuth` returns `{ state, sub, isGuest, scopes, email, name, isLoading,
refresh }`. Device info (`locale`, `userAgent`) stays on the separate
[`useUser`](/api-reference/use-user) hook — they answer different questions.

### Triggering login from the view

Login is **host-driven** and happens **out-of-band**: a view cannot open the
branded login page itself. What it *can* do is **trigger the challenge** — call
an `{ type: "oauth2" }` tool; the resulting 401/403 makes the host open its
OAuth flow. After the user signs in, call `refresh()` to re-read the identity.

```tsx
import { useAuth, useRequireAuth } from "enpilink/web";

function SaveButton() {
  const { state, refresh } = useAuth();
  // `save_favorite` is an oauth2-scoped server tool — calling it as a guest /
  // anonymous user returns a 401/403, which prompts the host to open OAuth.
  const requireAuth = useRequireAuth("save_favorite");
  return (
    <button onClick={() => requireAuth({ id: 1 }).then(refresh)}>
      {state === "authed" ? "Save" : "Sign in to save"}
    </button>
  );
}
```

<Note>
  Be honest with your users: the view only *triggers* the challenge. Whether and
  when the login window opens is entirely up to the host — there is nothing more
  the iframe can do to force it.
</Note>

### Configuring the upstream IdP

Set the upstream provider's endpoints + your registered client id. The client
**secret is env-only** (`ENPILINK_AUTH_CLIENT_SECRET`) — never in code, the DB,
or any API response. Auth stays **off** until you set this.

```bash
ENPILINK_AUTH=1
# This server (the AS/RS) — issuer is enpilink's own public origin:
ENPILINK_AUTH_ISSUER=https://my-app.example.com
ENPILINK_AUTH_AUDIENCE=https://my-app.example.com/mcp
ENPILINK_AUTH_JWKS_URL=https://accounts.google.com/.well-known/jwks   # the UPSTREAM JWKS (validates issued tokens)

# Upstream identity provider (example: Google):
ENPILINK_AUTH_UPSTREAM_CLIENT_ID=xxxx.apps.googleusercontent.com
ENPILINK_AUTH_UPSTREAM_AUTHORIZATION_URL=https://accounts.google.com/o/oauth2/v2/auth
ENPILINK_AUTH_UPSTREAM_TOKEN_URL=https://oauth2.googleapis.com/token
ENPILINK_AUTH_UPSTREAM_SCOPES="openid email profile"
ENPILINK_AUTH_CLIENT_SECRET=••••   # SECRET — env only

# The host connector callback URLs allowed as redirect targets:
ENPILINK_AUTH_REDIRECT_URIS="https://chatgpt.com/connector/oauth/<id>,https://claude.ai/api/mcp/auth_callback"
```

Programmatically, pass an `upstream` block (and optional `redirectUris`) in the
`auth` config:

```typescript
const server = new McpServer({ name: "my-app", version: "1.0.0" }, undefined, {
  auth: {
    enabled: true,
    issuer: "https://my-app.example.com",
    audience: "https://my-app.example.com/mcp",
    jwksUrl: "https://accounts.google.com/.well-known/jwks",
    redirectUris: ["https://claude.ai/api/mcp/auth_callback"],
    upstream: {
      clientId: "xxxx.apps.googleusercontent.com",
      authorizationUrl: "https://accounts.google.com/o/oauth2/v2/auth",
      tokenUrl: "https://oauth2.googleapis.com/token",
      scopes: ["openid", "email", "profile"],
    },
  },
});
```

### Authorization-server endpoints

When an upstream is configured, enpilink mounts (at the app root):

| Endpoint | Purpose |
| --- | --- |
| `/.well-known/oauth-authorization-server` | RFC 8414 AS metadata (discovery). Advertises `jwks_uri` in federating mode. |
| `/.well-known/oauth-protected-resource` | RFC 9728 PRM — lists this AS. |
| `/.well-known/jwks.json` | **Federating mode only** — enpilink's public signing key (validates the tokens it issues). |
| `/authorize` | Branded entry page → "Sign in" (upstream) or "Continue as guest" (federating). |
| `/authorize/upstream`, `/authorize/callback`, `/authorize/guest` | **Federating mode** — internal login/guest steps. |
| `/token` | Proxy mode: proxies the exchange upstream. Federating mode: validates PKCE + mints enpilink's token. |
| `/register` | Dynamic client registration — only if the upstream supports it. |

### Tracking users & sessions

On every successful token validation, enpilink upserts a **user** (keyed by the
`sub` claim) and records a **session** into the active storage adapter
(`memory` / `sqlite` / `postgres`). The write is best-effort — it never blocks
or slows a tool call (failures are swallowed and logged, like analytics). This
covers **both** authenticated users and **guests** (recorded with their
`guest:` sub).

Recorded: `sub`, issuer, client/host id, scopes, created/last-seen timestamps,
token expiry, and an **opaque token reference**. Sessions/users also carry an
`isGuest` flag (derived from the `guest:` sub prefix) so a dashboard can
distinguish guests from real users. Adapters expose
`upsertUser` / `recordSession` / `getSession` / `listSessions` / `listUsers`
(+ the optional `deleteSession` / `deleteUser` for revoke — see
[Storage → End-user auth tracking](/guides/storage#end-user-auth-tracking-optional)).

### Managing sessions in the console

The console's **Auth** tab is a dashboard for the sessions/users above. It lists
every tracked user and, under each, their recorded sessions — clearly
distinguishing **guest** vs **authed** (the `isGuest` flag), and showing `sub`,
issuer, client id, scopes, created / last-seen / token-expiry, and the opaque
`tokenRef`. It never displays a raw token or the signing key.

It is backed by a read/revoke API at `/__enpilink/auth` (same dev-open /
prod-admin-gated pattern as the observability + config APIs — in production it
sits behind the admin bearer token):

| Endpoint | Purpose |
| --- | --- |
| `GET /__enpilink/auth/sessions` | Recorded sessions (guests + authed), most recent first. |
| `GET /__enpilink/auth/users` | Tracked users, most recently seen first. |
| `DELETE /__enpilink/auth/sessions/:id` | **Revoke** a session — delete the row + denylist its access token. |
| `DELETE /__enpilink/auth/users/:sub` | Delete a user and cascade-revoke all their sessions. |

When storage is absent (auth off) every read returns `{ enabled: false }` with
an empty list — never a 500. None of these endpoints ever return a token or
secret; `tokenRef` is an opaque one-way hash, safe to expose for correlation.

<Warning>
**Revoke semantics.** enpilink's access tokens are stateless JWTs, so the
signature alone keeps them valid until `exp` (≤ 1 hour in federating mode).
Revoking a session does two things: (1) deletes the session/user row, and
(2) adds the token's reference to an **in-memory revocation denylist** that the
verifier checks on every request — so a revoked token is rejected (`401`)
immediately, not only at expiry. This makes revoke a *true* revocation.

The denylist (like the federation PKCE / refresh state) is **per-process and
in-memory**: a horizontally-scaled deployment needs a shared denylist (or sticky
sessions) for cross-instance revocation. Each entry auto-evicts at the token's
`exp`, so the set never grows unbounded. The revoke endpoint also drops the
federating provider's in-memory refresh token where it can.

Custom storage adapters made before this feature keep working: `deleteSession` /
`deleteUser` are **optional** on the interface, and the API reports the action as
unsupported (`501`) rather than failing when an adapter lacks them.
</Warning>

### Configuring auth in the console (Setup tab)

The **Auth → Setup** section is the place to configure end-user auth from the
console. It reuses the Configuration API and field components, so editing here is
identical to the Configuration tab — and the same guardrails apply.

It groups the auth keys for a setup-focused workflow:

1. **Enable & upstream identity provider** — `auth.enabled`, `auth.issuer`,
   `auth.audience`, `auth.jwksUrl`, the `auth.upstream.*` endpoints, and
   `auth.redirectUris`. These are **non-secret** and **editable** here; because
   the auth runtime is built once at boot, a change persists to the database and
   **takes effect after a restart** (shown with a "requires restart" badge). When
   auth is off, this is also where you turn it on.
2. **Secrets** — `auth.signingKey` and `auth.clientSecret`. These are
   **environment-only**: rendered masked + read-only with a "set via `ENV_VAR`"
   hint. The console can never write them (the config API returns `403` on any
   `PUT`), and they are never returned in plaintext. Setting `auth.signingKey`
   switches enpilink into federating mode (it mints its own tokens + enables
   guest sign-in).
3. **Login-page branding** — see below — with a **live preview** of the branded
   login page.

<Warning>
The two auth **secrets stay environment-only**. The Setup screen renders them
read-only and the config API rejects any attempt to write them — they can only
be set via `ENPILINK_AUTH_SIGNING_KEY` / `ENPILINK_AUTH_CLIENT_SECRET`. Branding
and the non-secret upstream config are safe to edit from the console.
</Warning>

### Branding the login page

The server-rendered login page (the "Sign in / Continue as guest" interstitial)
is customizable, Auth0-style, via four **non-secret** keys:

| Key | Env var | Effect |
| --- | --- | --- |
| `auth.branding.appName` | `ENPILINK_AUTH_BRANDING_APP_NAME` | Heading ("Sign in to continue to *name*"). Defaults to the MCP server name. |
| `auth.branding.logoUrl` | `ENPILINK_AUTH_BRANDING_LOGO_URL` | A logo image (http/https only) shown above the heading. Defaults to a monogram. |
| `auth.branding.accentColor` | `ENPILINK_AUTH_BRANDING_ACCENT_COLOR` | Accent color (CSS hex, e.g. `#3fb6a8`) for the button + monogram. Defaults to enpilink teal. |
| `auth.branding.tagline` | `ENPILINK_AUTH_BRANDING_TAGLINE` | Short copy under the heading. Defaults to the enpilink sign-in copy. |

Set them via env/file or edit them in the **Auth → Setup → Login-page branding**
group (they take effect after a restart, like the other auth keys). The login
page consumes them and **falls back to enpilink defaults** for any value that is
unset. Branding is **presentational only** — values are escaped (text) or
hex/URL-validated (color/logo), so they can never inject markup or CSS, and they
never change the OAuth flow (the redirect allowlist, the `HttpOnly` state-binding
cookie, and PKCE are untouched).

### Tokens at rest

enpilink **never stores the raw access or refresh token** (upstream or its own).
The session row holds only a **one-way SHA-256 reference** (`tokenRef`) used to
correlate and de-duplicate sessions. Because no secret token is persisted, there
is nothing at rest to decrypt or leak — no encryption key is required for
session storage. `ENPILINK_AUTH_SIGNING_KEY` (federating mode) signs **issued**
tokens; it is held in memory only and never persisted.

## Secrets

Any auth secret (`ENPILINK_AUTH_SIGNING_KEY`, `ENPILINK_AUTH_CLIENT_SECRET`)
is **env-only**: shown masked + read-only in the Configuration UI, never
persisted to the database, never returned in plaintext by any API — the same
treatment as the admin token. The upstream **client id** and endpoint URLs are
non-secret config and may be set via file/env normally.
