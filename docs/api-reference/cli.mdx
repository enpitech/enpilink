enpilink provides two CLI tools: the main `enpilink` CLI for development and production workflows, and `create-enpilink` for bootstrapping new projects.

## enpilink CLI

The main CLI is available as both `enpilink` and `enpi` commands after installing `enpilink` in your project.

### enpilink dev

Start the development server with hot module reloading and the Console.

```bash
enpilink dev
```

**What it does:**
- Starts a development server at `http://localhost:3000/`
- Serves the Console UI at `http://localhost:3000/`
- Exposes MCP server at `http://localhost:3000/mcp`
- Enables file watching with automatic server restart
- Enables Hot Module Reloading (HMR) for views

#### Flags

| Flag | Description |
|------|-------------|
| `-p, --port <number>` | Port to run the server on. Defaults to `3000`, or the next available port if `3000` is in use. |
| `--tunnel` | Open an account-free [srv.us](https://srv.us) SSH tunnel for remote testing (no login, no signup). |
| `--open` | Open the Console in the browser when the server is ready (off by default). |
| `--mock` | Seed the Dashboard with deterministic demo analytics for demos/screenshots. Implies analytics-on and in-memory storage for the session — never touches disk and vanishes on exit. Dev-only; ignored in production. |
| `-v, --verbose` | Show tunnel logs. |

You can also set the port via the `PORT` environment variable:

```bash
PORT=8080 enpilink dev
```

To have the Console open automatically on every `dev` run, set `ENPILINK_OPEN=true`:

```bash
# one-off
ENPILINK_OPEN=true enpilink dev

# persistent (add to your shell profile)
export ENPILINK_OPEN=true
```

### enpilink build

Build your views and MCP server for production deployment.

```bash
enpilink build
```

**What it does (in order):**

1. **Scanning views** - Discovers your view components under `src/views/`
2. **Compiling server** - Transpiles your TypeScript server with `tsc`
3. **Rewriting path aliases** - Rewrites server `@/…` imports in the emitted JS to relative paths (so `node dist/__entry.js` resolves them — you do **not** need `tsc-alias` in your own scripts)
4. **Building views** - Compiles your React views with Vite
5. **Emitting** the manifest module, the `dist/__entry.js` entry wrapper, and the Cloudflare / Vercel deployment artifacts

The output is placed in the `dist/` directory, ready for production deployment.

### enpilink codegen

Regenerate the view/tool type declarations (`.enpilink/views.d.ts`) without running a full build.

```bash
enpilink codegen
```

**What it does:**
- Scans your view components under `src/views/`
- Writes `.enpilink/views.d.ts` (creating the `.enpilink/` directory if missing), which augments `enpilink/server`'s `ViewNameRegistry` with your view names so view-name generics like `useToolInfo<"checkout">()` resolve

This is the same artifact `enpilink build` and `enpilink dev` produce, factored out as a fast, standalone step. It is idempotent, deterministic (view names are sorted), and always succeeds — a fresh project with **zero views** still emits a valid empty registry.

<Note>
Generated projects wire this into the `typecheck` script (`enpilink codegen && tsc --noEmit`) so a freshly scaffolded project type-checks cleanly **without** needing a prior build. If you run `tsc --noEmit` directly on a fresh checkout (before any `build`/`dev`/`codegen`), view-name generics fall back to `never` until `.enpilink/views.d.ts` exists — run `enpilink codegen` first.
</Note>

`.enpilink/` is generated output and should stay gitignored (the scaffolded `.gitignore` already excludes it).

### enpilink start

Start the production server.

```bash
enpilink start
```

**Requirements:**
- Must run `enpilink build` first
- Requires `dist/__entry.js` (the entry wrapper, which primes the Vite manifest so hashed asset URLs resolve). Falls back to `dist/server.js` if the wrapper is absent.

**Flags**

| Flag | Description |
|------|-------------|
| `-p, --port <number>` | Port to run the server on. Defaults to `3000`, or the next available port if `3000` is in use. |
| `--admin` | Enable the production admin plane (Console Dashboard + Configuration + observability API) behind bearer auth. Off by default. Requires `ENPILINK_ADMIN_TOKEN` to be set — the server refuses to start without it. Also enableable via `ENPILINK_ADMIN=1`. |

**What it does:**
- Runs the compiled server (`node dist/__entry.js`)
- Sets `NODE_ENV=production`
- Listens on the port from the `__PORT` environment variable (default `3000`)
- Serves the MCP endpoint at `http://localhost:3000/mcp` and pre-built view assets from `/assets`

<Note>
The production entry reads the **`__PORT`** environment variable (not `PORT`), defaulting to `3000`. `enpilink start` sets it for you; if you run `node dist/__entry.js` directly, set `__PORT` yourself (e.g. `__PORT=8080 node dist/__entry.js`).
</Note>

## create-enpilink

A standalone CLI for bootstrapping new enpilink projects.

```bash
npm create enpilink@latest [directory]
```

Or with other package managers:

<CodeGroup>
```bash npm
npm create enpilink@latest
```
```bash pnpm
pnpm create enpilink@latest
```
```bash yarn
yarn create enpilink
```
```bash bun
bun create enpilink
```
</CodeGroup>

### Arguments

| Argument | Description |
|----------|-------------|
| `[directory]` | Target directory for the new project. If not specified, you'll be prompted to enter a project name. Use `.` to create in the current directory. |

### Options

| Option | Description |
|--------|-------------|
| `-h, --help` | Show help message |
| `--overwrite` | Remove existing files in target directory without prompting |
| `--immediate` | Install dependencies and start the development server automatically after scaffolding |

### Examples

**Create a new project interactively:**

```bash
npm create enpilink@latest
```

**Create a project in a specific directory:**

```bash
npm create enpilink@latest my-app
```

**Create in current directory, overwrite existing files, and start immediately:**

```bash
npm create enpilink@latest . --overwrite --immediate
```

### What it creates

The scaffolder copies a starter template that includes:

- `server/index.ts` - MCP server with example tools
- `web/` - React views with example components
- `package.json` - Pre-configured with all dependencies
- `tsconfig.json` files - TypeScript configuration
- `vite.config.ts` - Vite configuration for view bundling
- `nodemon.json` - File watching configuration for development

## Package Manager Scripts

When you create a new enpilink project, the `package.json` includes these scripts:

| Script | Command | Description |
|--------|---------|-------------|
| `dev` | `enpilink dev` | Start development server |
| `build` | `enpilink build` | Build for production |
| `start` | `enpilink start` | Start production server |
| `typecheck` | `enpilink codegen && tsc --noEmit` | Regenerate view type declarations, then type-check (works on a fresh project with no prior build) |

Run them with your package manager:

<CodeGroup>
```bash npm
npm run dev
npm run build
npm start
```
```bash pnpm
pnpm dev
pnpm build
pnpm start
```
```bash yarn
yarn dev
yarn build
yarn start
```
```bash bun
bun dev
bun build
bun start
```
```bash deno
deno task dev
deno task build
deno task start
```
</CodeGroup>
